什么是一致性哈希

一致性哈希（Consistent Hashing）是一种特殊的哈希算法，由 MIT 的 David Karger 等人在 1997 年提出，主要用于分布式缓存系统中。

核心特点

特性 说明
分散性 数据均匀分布在各个节点上
单调性 添加/删除节点时，只影响部分数据
平衡性 各节点负载相对均衡
低迁移率 节点变化时，数据迁移量最小化

为什么需要一致性哈希

传统哈希的问题

在传统哈希方案中，数据映射公式为：

node_index = hash(key) % node_count

问题场景：假设有 3 个节点，1000 条数据

初始状态：
节点 A: 333 条数据
节点 B: 333 条数据
节点 C: 334 条数据

当节点 C 宕机后：
node_index = hash(key) % 2  ← 除数变了！

结果：几乎所有数据都需要重新映射
- 节点 A: 500 条数据（新增 167 条）
- 节点 B: 500 条数据（新增 167 条）
- 缓存命中率暴跌！

一致性哈希的优势

一致性哈希方案：
当节点 C 宕机后：
- 只有 C 节点的数据需要迁移
- 其他节点的数据基本不受影响
- 缓存命中率保持在较高水平

工作原理

1. 哈希环

一致性哈希将整个哈希值空间组织成一个虚拟的圆环：

                    哈希环 (0 ~ 2^32-1)
                    
                         0 (2^32)
                           ↑
                          / \
                         /   \
              节点 A ←  •     •  → 节点 B
                       |       |
                       |       |
              数据 X ←  •       •  → 数据 Y
                         \   /
                          \ /
                           ↓
                        节点 C

2. 数据映射规则

┌─────────────────────────────────────────────────────────┐
│  映射规则：顺时针查找第一个节点                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 计算数据的哈希值，映射到环上                          │
│  2. 沿顺时针方向查找，遇到的第一个节点即为存储节点         │
│  3. 如果该节点宕机，继续顺时针找下一个节点                │
│                                                         │
└─────────────────────────────────────────────────────────┘

3. 节点变化影响

添加节点 D：

        添加前                    添加后
        
           A                         A
         /   \                     /   \
        •     •  B               •  D  •  B
        |     |       →          | /   |
        |     |                  |/    |
        •     •  C               •     •  C
        
影响范围：只有 B→C 之间的数据需要迁移到 D

虚拟节点

数据倾斜问题

当物理节点较少时，数据分布可能不均匀：

哈希环示例：

    节点 A
       ↑
      / \
     /   \
    •     •  
    |     |  ← 节点 A 负责大部分数据
    |     |
    •     •
     \   /
      \ /
       ↓
    节点 B

结果：节点 A 负载 80%，节点 B 负载 20%

解决方案：虚拟节点

为每个物理节点创建多个虚拟节点：

物理节点 → 虚拟节点映射

节点 A → A1, A2, A3, ..., A100
节点 B → B1, B2, B3, ..., B100
节点 C → C1, C2, C3, ..., C100

哈希环：

    A1
   /  \
  C5   B2
 /      \
B8       A3
 \      /
  C2   A7
   \  /
    B5

结果：数据均匀分布，每个物理节点负责约 33% 的数据

虚拟节点数量建议

场景 虚拟节点数 说明
小型集群（<10 节点） 100-200 保证数据均匀分布
中型集群（10-100 节点） 50-100 平衡性能和分布
大型集群（>100 节点） 20-50 减少环上节点数量

代码实现

Python 实现示例

import hashlib
import bisect
from typing import Dict, List, Any

class ConsistentHash:
    """一致性哈希实现（支持虚拟节点）"""
    
    def __init__(self, nodes: List[str] = None, virtual_nodes: int = 100):
        """
        初始化一致性哈希环
        
        Args:
            nodes: 初始节点列表
            virtual_nodes: 每个物理节点的虚拟节点数量
        """
        self.virtual_nodes = virtual_nodes
        self.ring: Dict[int, str] = {}  # 哈希值 → 节点名
        self.sorted_keys: List[int] = []  # 排序后的哈希值列表
        
        # 添加初始节点
        if nodes:
            for node in nodes:
                self.add_node(node)
    
    def _hash(self, key: str) -> int:
        """计算 key 的哈希值（32 位）"""
        return int(hashlib.md5(key.encode()).hexdigest(), 16) % (2 ** 32)
    
    def add_node(self, node: str) -> None:
        """添加节点（包括虚拟节点）"""
        for i in range(self.virtual_nodes):
            virtual_key = f"{node}#{i}"
            hash_value = self._hash(virtual_key)
            
            self.ring[hash_value] = node
            bisect.insort(self.sorted_keys, hash_value)
    
    def remove_node(self, node: str) -> None:
        """移除节点"""
        for i in range(self.virtual_nodes):
            virtual_key = f"{node}#{i}"
            hash_value = self._hash(virtual_key)
            
            if hash_value in self.ring:
                del self.ring[hash_value]
                self.sorted_keys.remove(hash_value)
    
    def get_node(self, key: str) -> str:
        """获取 key 应该存储的节点"""
        if not self.ring:
            raise Exception("哈希环为空")
        
        hash_value = self._hash(key)
        
        # 二分查找顺时针第一个节点
        idx = bisect.bisect(self.sorted_keys, hash_value)
        if idx >= len(self.sorted_keys):
            idx = 0
        
        return self.ring[self.sorted_keys[idx]]
    
    def get_distribution(self) -> Dict[str, int]:
        """获取当前数据分布统计"""
        distribution = {}
        for node in self.ring.values():
            distribution[node] = distribution.get(node, 0) + 1
        return distribution


# 使用示例
if __name__ == "__main__":
    # 创建一致性哈希环
    ch = ConsistentHash(nodes=["A", "B", "C"], virtual_nodes=100)
    
    # 模拟数据存储
    keys = [f"data_{i}" for i in range(1000)]
    storage = {node: [] for node in ["A", "B", "C"]}
    
    for key in keys:
        node = ch.get_node(key)
        storage[node].append(key)
    
    # 输出分布情况
    print("数据分布：")
    for node, data_list in storage.items():
        print(f"  节点 {node}: {len(data_list)} 条数据 ({len(data_list)/len(keys)*100:.1f}%)")
    
    # 模拟节点扩容
    print("\n添加节点 D...")
    ch.add_node("D")
    
    # 重新计算分布
    new_storage = {node: [] for node in ["A", "B", "C", "D"]}
    migrated_count = 0
    
    for key in keys:
        node = ch.get_node(key)
        new_storage[node].append(key)
    
    print("新数据分布：")
    for node, data_list in new_storage.items():
        print(f"  节点 {node}: {len(data_list)} 条数据 ({len(data_list)/len(keys)*100:.1f}%)")

优缺点分析

✅ 优点

优点 说明
低迁移率 节点变化时，只影响部分数据（约 1/N）
高可用 节点故障自动转移，无需人工干预
易扩展 添加/删除节点简单，不影响整体服务
去中心化 无需中心协调节点，各节点独立
负载均衡 配合虚拟节点，数据分布均匀

❌ 缺点

缺点 说明 解决方案
数据倾斜 节点少时分布不均 使用虚拟节点
查询复杂度 需要 O(log N) 查找 使用平衡树/跳表
实现复杂 比普通哈希复杂 使用成熟库
热点数据 无法解决单点热点 配合其他策略

何时使用一致性哈希

适合场景：

  • ✅ 分布式缓存系统
  • ✅ 需要频繁扩缩容的集群
  • ✅ 对缓存命中率要求高的场景
  • ✅ P2P 网络和 DHT 系统

不适合场景：

  • ❌ 节点数量固定的小型系统
  • ❌ 需要严格数据一致性的场景
  • ❌ 数据量极小，单节点即可承载
