什么是一致性哈希

一致性哈希（Consistent Hashing）是一种特殊的哈希算法，由 MIT 的 David Karger 等人在 1997 年提出，主要用于分布式缓存系统中。

核心特点

🎯

分散性

数据均匀分布在各个节点上

📈

单调性

添加/删除节点时，只影响部分数据

⚖️

平衡性

各节点负载相对均衡

🔄

低迁移率

节点变化时，数据迁移量最小化

为什么需要一致性哈希

传统哈希的问题

在传统哈希方案中，数据映射公式为：

哈希公式
node_index = hash(key) % node_count

问题场景：假设有 3 个节点，1000 条数据

❌

传统哈希 - 初始状态

节点 A
333 条数据
节点 B
333 条数据
节点 C
334 条数据
→ 节点 C 宕机 →
⚠️

传统哈希 - 宕机后

节点 A
500 条数据
节点 B
500 条数据
❌ 几乎所有数据都需要重新映射！缓存命中率暴跌！

一致性哈希的优势

✅

一致性哈希方案

✓ 只有 C 节点的数据需要迁移
✓ 其他节点的数据基本不受影响
✓ 缓存命中率保持在较高水平

2. 数据映射规则

📐

映射规则：顺时针查找第一个节点

1 计算数据的哈希值，映射到环上
2 沿顺时针方向查找，遇到的第一个节点即为存储节点
3 如果该节点宕机，继续顺时针找下一个节点

虚拟节点

数据倾斜问题

当物理节点较少时，数据分布可能不均匀：

节点 A 负载 80%
节点 B 负载 20%

解决方案：虚拟节点

为每个物理节点创建多个虚拟节点：

🔵

节点 A

A1, A2, A3, ..., A100
🟢

节点 B

B1, B2, B3, ..., B100
🟡

节点 C

C1, C2, C3, ..., C100
✅
数据均匀分布，每个物理节点负责约 33% 的数据

代码实现

Python 实现示例

Python 一致性哈希实现（支持虚拟节点） 
import hashlib
import bisect
from typing import Dict, List, Any


class ConsistentHash:
    """一致性哈希实现（支持虚拟节点）"""


def __init__(self, nodes: List[str] = None, virtual_nodes: int = 100):
    self.virtual_nodes = virtual_nodes
    self.ring: Dict[int, str] = {}
    self.sorted_keys: List[int] = []
    
    if nodes:
        for node in nodes:
            self.add_node(node)

def _hash(self, key: str) -> int:
    """计算 key 的哈希值（32 位）"""
    return int(hashlib.md5(key.encode()).hexdigest(), 16) % (2 ** 32)

def add_node(self, node: str) -> None:
    """添加节点（包括虚拟节点）"""
    for i in range(self.virtual_nodes):
        virtual_key = f"{node}#{i}"
        hash_value = self._hash(virtual_key)
        self.ring[hash_value] = node
        bisect.insort(self.sorted_keys, hash_value)

def remove_node(self, node: str) -> None:
    """移除节点"""
    for i in range(self.virtual_nodes):
        virtual_key = f"{node}#{i}"
        hash_value = self._hash(virtual_key)
        if hash_value in self.ring:
            del self.ring[hash_value]
            self.sorted_keys.remove(hash_value)

def get_node(self, key: str) -> str:
    """获取 key 应该存储的节点"""
    if not self.ring:
        raise Exception("哈希环为空")
    
    hash_value = self._hash(key)
    idx = bisect.bisect(self.sorted_keys, hash_value)
    if idx >= len(self.sorted_keys):
        idx = 0
    
    return self.ring[self.sorted_keys[idx]]

def get_distribution(self) -> Dict[str, int]:
    """获取当前数据分布统计"""
    distribution = {}
    for node in self.ring.values():
        distribution[node] = distribution.get(node, 0) + 1
    return distribution

if name == "main":

应用场景

🗄️

分布式缓存系统

Client → 一致性哈希 → Redis 集群
  • Redis Cluster：使用哈希槽（16384 个槽）
  • Memcached：客户端实现一致性哈希
  • Codis：Twitter 开源的 Redis 代理
🗃️

分布式数据库

用户 ID → 一致性哈希 → 数据库分片
  • Cassandra：使用一致性哈希进行数据分区
  • DynamoDB：基于一致性哈希的键空间分区
  • MongoDB：Sharding 支持一致性哈希策略
⚖️

负载均衡

请求 → 一致性哈希 → 后端服务器
  • Session 保持：同一用户路由到同一服务器
  • 弹性伸缩：服务器扩缩容时影响最小化
🌐

P2P 网络

Chord、Kademlia 等 DHT 协议都基于一致性哈希
  • 去中心化：节点自由加入/离开
  • 高效定位：数据定位效率 O(log N)
  • 抗故障：节点故障不影响整体网络

优缺点分析

✅

优点

🔄
低迁移率

节点变化时，只影响部分数据（约 1/N）

🛡️
高可用

节点故障自动转移，无需人工干预

📈
易扩展

添加/删除节点简单，不影响整体服务

🔗
去中心化

无需中心协调节点，各节点独立

⚖️
负载均衡

配合虚拟节点，数据分布均匀

⚠️

缺点 & 解决方案

数据倾斜 节点少时分布不均
解决 使用虚拟节点
查询复杂度 需要 O(log N) 查找
解决 使用平衡树/跳表
实现复杂 比普通哈希复杂
解决 使用成熟库
热点数据 无法解决单点热点
解决 配合其他策略

总结

核心要点

1️⃣
哈希环 将哈希空间组织成环形结构
2️⃣
顺时针查找 数据映射到顺时针第一个节点
3️⃣
虚拟节点 解决数据倾斜问题
4️⃣
低迁移率 节点变化只影响部分数据

何时使用一致性哈希

✅

适合场景

  • 🗄️ 分布式缓存系统
  • 📈 需要频繁扩缩容的集群
  • 🎯 对缓存命中率要求高的场景
  • 🌐 P2P 网络和 DHT 系统
❌

不适合场景

  • 🔒 节点数量固定的小型系统
  • 📋 需要严格数据一致性的场景
  • 💾 数据量极小，单节点即可承载
